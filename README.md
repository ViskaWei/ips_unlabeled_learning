## 💡 为什么需要这个系统？

探索性研究的常见痛点：

| 问题 | 症状 |
|------|------|
| **结论不可追溯** | 实验跑了很多，但不知道为什么得出这个结论 |
| **决策无依据** | "感觉应该加数据"——缺乏量化的 Go/No-Go 标准 |
| **知识不复用** | 每次新项目从零开始，重复踩坑 |
| **信息散落** | 笔记在 Notion、代码在 GitHub、讨论在 Slack |

**本系统的解法**：

```
假设树拆问题 → 每个假设对应 MVP 实验 → 提前写好验收阈值 + 止损规则
      ↓                                              ↓
  实验产出标准化报告 ← ← ← ← 关键数字回流 ← ← ← ← Gate 决策
      ↓
  沉淀为可复用的设计原则
```

**核心价值**：在不确定性很强的任务里，稳定地推进到可交付的结果，并让团队任何人都能追溯"为什么走这条路"。

### 文档层级

| 层级 | 文件 | 职责 | 更新时机 |
|------|------|------|---------|
| 1 | `session.md` | GPT 脑暴对话 → 结构化实验候选 | 讨论后 |
| 2a | `hub.md` | 问题树、假设、战略导航、设计原则 | 认知改变时 |
| 2b | `roadmap.md` | MVP 列表、进度追踪、数值结果 | 实验状态变更时 |
| 3 | `exp.md` | 单实验详细记录 | 实验完成时 |
| 4 | `card.md` | 跨实验知识卡片 | 产出可复用结论时 |
| 5 | `next_steps.md` | 日常待办 | 每日 |

### Hub vs Roadmap 职责边界

| 维度 | Hub (智库) | Roadmap (执行) |
|------|-----------|---------------|
| **核心问题** | "我们知道了什么？往哪走？" | "计划跑哪些实验？进度如何？" |
| **内容** | 假设树、洞见汇合、战略推荐、设计原则 | MVP 列表、Gate 定义、进度看板、数值结果 |

## 🚦 Decision Gate（资源分配决策）

不再"凭感觉"决定下一步，用 3 个量化信号做裁决：

```
Q: 下一步该投什么？
│
├── Gate-1: 理论上限 (Fisher/CRLB)
│   ├── If headroom > 20% → 继续投模型
│   └── If headroom < 5%  → 转向其他任务
│
├── Gate-2: Scaling 趋势
│   ├── If Δ > 10% per 3× data → 继续投数据
│   └── If Δ < 5% per 3× data  → 边际递减，投模型
│
└── Gate-3: 结构红利 (Oracle 实验)
    ├── If ΔR² > 0.05 → 投结构 (MoE/分域)
    └── Else          → 结构无效，换方向
```

## 🚀 快速开始

### 1. 使用此模板创建新项目

点击 GitHub 页面上的 **"Use this template"** 按钮，或：

```bash
gh repo create my-research-project --template YOUR_USERNAME/AI_Research_Template
cd my-research-project
```

### 2. 配置项目

编辑 `.cursorrules` 文件，更新项目特定信息：
- 报告输出路径
- 项目背景和目标
- 常用指标和实验变量

### 3. 开始第一个实验主题

```bash
# 在 Cursor 中输入
n [实验描述]
```

系统会自动创建：
- `experiments/[topic]/[topic]_hub.md` — 智库导航
- `experiments/[topic]/[topic]_roadmap.md` — 实验追踪
- `experiments/[topic]/exp_*.md` — 实验报告


## 🎯 MVP 设计规范

每个 MVP 必须满足：

| 要素 | 要求 | 示例 |
|------|------|------|
| **精确假设** | 一句话可证伪 | "lr=0.1 比 lr=0.05 提升 ≥5%" |
| **最小配置** | 只改一个变量 | 固定 data/model，只变 lr |
| **量化验收** | If-Then 规则 | If ΔR² ≥ 0.05 → 接受 |
| **明确止损** | 失败后的行动 | 拒绝 → 关闭此方向 |

## 📚 设计理念

本系统基于以下原则构建：

1. **假设驱动**：大问题拆成可证伪的小假设
2. **最小验证**：每个假设对应一个最低成本的 MVP
3. **量化决策**：所有 Gate 都有数字阈值，不靠"感觉"
4. **知识回流**：实验结论自动沉淀到 Hub 和设计原则
5. **可追溯性**：任何结论都能追溯到具体实验和数据
